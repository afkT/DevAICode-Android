# AI 高效开发工作流 · 知识库

本知识库提炼自 `ai/guides` 内研究、规划、批注、实现四阶段指南，供后续任务快速查阅与引用。

---

## 1. 工作流总览

| 阶段 | 目的 | 产出物 | 何时可缩短/跳过 |
|------|------|--------|------------------|
| **研究** | 在设计与写码前深度理解相关代码 | `research.md` | 单行/单文件级琐碎改动 |
| **规划** | 将研究转化为可评审的实施计划 | `plan.md` | 无（规划前必须先有研究） |
| **批注循环** | 在计划中注入判断与约束，迭代至通过 | 更新后的 `plan.md` + 待办列表 | 无 |
| **实现** | 按通过计划执行，你监督与纠偏 | 可运行代码 | 无 |

**核心原则**：在未审阅并认可**书面** research 与 **书面** plan 之前，不要让 AI 写代码。计划即契约；批注是你把关的环节。

**文件约定**：
- 研究：`docs/research/`，编号命名如 `001-xxx.md`
- 计划：`docs/plans/`，编号命名如 `001-xxx.md`

## 2. 研究阶段 (Research)

### 2.1 何时使用
- 新功能/新模块、重大或架构级改动
- 需要先吃透某块复杂逻辑再改
- 代码库演进后需要刷新理解

### 2.2 何时缩短或跳过
- 单行配置、单文件重命名等**琐碎改动**可快速扫读或跳过完整 research
- 凡涉及架构、数据流、多文件：**不要跳过**

### 2.3 强调词（避免 AI 泛读）
- "deeply" / "深入"
- "in great detail" / "详尽"
- "intricacies" / "细节"
- "go through everything" / "通读"

### 2.4 research.md 应包含
- 系统架构与数据流
- 关键组件及职责
- 依赖与关系
- 既有模式与约定
- 潜在问题或 Bug
- 边界情况与特殊行为

### 2.5 常用提示词模板
- 通用：`深度阅读 [路径]，理解其如何工作、做什么及所有特性。完成后在 [路径]/research.md 中写详细报告。`
- 系统专项：`深入研究 [系统名]，理解其细节，并写一份 research.md，包含关于 [系统] 运作方式的一切。`
- 迭代更新：`代码库已有变更，审阅变更并据此更新 research.md，关注对当前系统的影响。`

### 2.6 研究阶段检查清单
- [ ] 使用了强调词
- [ ] 生成了 research.md
- [ ] 你已审阅并确认结论
- [ ] 识别了既有模式与约定
- [ ] 记录了潜在问题与边界情况
- [ ] 存放在约定目录（如 `docs/research/`）

## 3. 规划阶段 (Planning)

### 3.1 何时使用
- 研究完成后、任何实现之前
- 需要与人对齐方案或评审

### 3.2 plan.md 应包含
- **详细说明**：思路、理由、考虑过的替代方案及为何不选
- **代码片段**：真实改动示例、前后对比、与现有代码的集成点
- **文件清单**：要修改/新增/删除的文件及目录结构
- **考虑与取舍**：性能、安全、兼容、迁移、测试

### 3.3 计划文档模板（结构）
```markdown
# 计划：[功能名]
## 概述 | ## 方案 | ## 需修改的文件 | ## 新增文件
## 实现细节（代码片段）| ## 考虑（性能/安全/兼容/测试）| ## 取舍
```

### 3.4 常用提示词模板
- 功能：`我想实现 [功能描述]，达成 [业务结果]。写一份详细 plan.md，含实现思路与代码片段。`
- 重构：`[需求变更]。写详细 plan.md，先读源码再建议，基于真实代码库。`
- 参考实现：`[某项目] 中是这样做的 [代码/链接]，写 plan.md 说明我们如何采用类似做法。`

### 3.5 规划阶段检查清单
- [ ] 计划含详细思路与代码片段
- [ ] 列出所有要改/要建的文件
- [ ] 记录了考虑与取舍
- [ ] 基于真实代码库（非假设）
- [ ] 含测试策略
- [ ] 可进入批注循环（足够具体可评审）

## 4. 批注循环 (Annotation Cycle)

### 4.1 循环步骤
1. AI 写出 plan.md  
2. 你在编辑器中打开 plan.md，在有问题处**加内联批注**  
3. 把文档发回 AI，并说明：**「根据所有批注更新文档，不要开始实现。」**  
4. 重复 2–3 直到满意（通常 1–6 轮）  
5. 请求在**同一文档**中生成**详细待办列表**，再进入实现

### 4.2 批注类型
- **领域知识**：工具/约定（如 "用 drizzle:generate 做迁移"、"用项目内 logger"）
- **纠正假设**：如 "应是 PATCH 不是 PUT"、"此处已鉴权，不要加 auth 中间件"
- **否决方案**：如 "删掉整段，这里不需要缓存"、"用轮询不要用 WebSocket"
- **补充上下文**：如 "队列消费者已负责重试，这里不必再写重试"
- **重定向整段**：如 "可见性应在 list 上而非 item 上，重写该节"
- **增加约束**：如 "不可选"、"必须兼容 v1 API"、"必须支持离线"

### 4.3 迭代策略
- 通常 1–6 轮；停下的条件：顾虑都解决、方案符合需求、你认可
- 必须带 **「不要实现」/「don't implement yet」**，否则 AI 容易直接写代码

### 4.4 批准后必做
请求待办列表的提示词示例：  
`在计划中增加一份详细待办列表，包含所有阶段与单项任务，不要实现。`

### 4.5 批注循环检查清单
- [ ] 在编辑器中打开并通读 plan
- [ ] 对关键问题加了具体、可执行的批注
- [ ] 发回时包含「不要实现」
- [ ] 每轮更新后都再审阅
- [ ] 已请求并确认待办列表后再进入实现

## 5. 实现阶段 (Implementation)

### 5.1 何时使用
- 计划已通过、待办列表已写入 plan 之后

### 5.2 标准实现提示词（可适配技术栈）
```
全部实现。每完成一个任务或阶段，在计划文档中将其标为已完成。不要停，直到所有任务和阶段完成。不要加多余注释或 KDoc。持续运行 lint 和编译以确认未引入新问题。
```

根据项目实际情况可替换为 `./gradlew lint`、`./gradlew compileDebugKotlin`、单元测试等，原则相同：**边做边校验**。

### 5.3 你的角色转变
- 规划时：**架构决策**
- 实现时：**监督与纠偏**，用**简短、直接**的修正

### 5.4 反馈类型示例
- 缺实现：`你没有实现 xxx 函数/模块。`
- 放错层：`这应放在 data 层，不是 domain 层。`
- 方式不对：`用 Flow，不要用 LiveData。` / `用现有 Repository，不要新建。`
- 风格：`遵循现有命名/格式。`  
前端可配合截图 + 参照现有页面：`这个列表要和用户列表长得一样，同一表头、分页、行高。`

### 5.5 何时回滚
- 方向根本错误、问题太多、方案与需求不符、发现计划有缺陷 → **回滚并收窄范围**，比在错误基础上打补丁更高效。

### 5.6 实现阶段检查清单
- [ ] 计划已批准且含待办列表
- [ ] 使用了标准实现提示词
- [ ] AI 在 plan 中勾选完成项
- [ ] 持续跑 lint/编译/测试
- [ ] 你用简短句做纠偏，避免 micromanage
- [ ] 所有任务完成后做一次验证与测试

## 6. 最佳实践与常见坑

### 最佳实践
- 研究请求里用强调词；只认书面 research + plan
- 多轮短批注优于一次长批注
- 以你的审阅和批注为门禁，不给 AI 完全自主权
- 尽量**单次长会话**完成研究→规划→批注→实现，保持上下文

### 常见坑
- **跳过研究** → 计划错、返工多
- **跳过批注** → 首版计划常有问题，批注是低成本纠偏
- **实现阶段 micromanage** → 说「全部实现」后让 AI 按任务跑，按批次反馈
- **不做持续校验** → 边做边跑 lint/编译/测试

## 7. 快速参考：各阶段一句话 + 产出

| 阶段 | 你一句话 | 产出 |
|------|----------|------|
| 研究 | 深度读 [范围]，写 research.md | `docs/research/xxx.md` |
| 规划 | 基于 research 写 plan.md，含方案与代码片段 | `docs/plans/xxx.md` |
| 批注 | 我加了批注，按批注更新文档，不要实现 | 更新后的 plan.md |
| 待办 | 在计划里加详细待办列表，不要实现 | plan 中的 Todo 列表 |
| 实现 | 全部实现，完成项在计划里打勾，持续 lint/编译 | 可运行代码 |

**说明**：表中用方括号包起来的如 `[范围]`、`xxx` 是**占位符**，需你替换成实际内容。例如「深度读 [范围]」即：把 [范围] 换成你要研究的目录或模块名（如 `app/data/user/`、`app/domain/user/`），不是文档内的引用链接。
