# 完整工作流程示例：用户搜索功能

本文档通过**用户搜索功能**示例，演示 AI 高效开发工作流程的四个阶段；重点体现**流程与步骤**，仅保留核心代码。

---

## 功能需求

**目标**：实现用户搜索功能，允许用户通过用户名或邮箱搜索其他用户。

**要求**：
- 支持按用户名和邮箱搜索
- 搜索不区分大小写
- 支持分页，每页 20 条结果
- 显示用户头像、用户名、邮箱
- 需要处理离线状态

---

## 阶段一：研究阶段 (Research)

**目的**：在改任何代码前，让 AI 深度理解用户相关模块（数据层、领域层、API、数据库）。

### 步骤 1：发出研究提示词

```
深度阅读 app/data/user/ 和 app/domain/user/，理解用户系统如何工作、做什么以及所有细节。特别关注：
1. UserRepository 的结构和现有方法
2. User 模型的字段定义
3. 现有 API 接口和数据流
4. 数据库表结构

完成后在 docs/research/001-user-search.md 中写一份详细的学习与发现报告。
```

### 步骤 2：AI 产出 research.md 要点

- **数据层**：UserRepositoryImpl、UserDao、UserDto、UserEntity 职责与关系
- **领域层**：UserRepository 接口、User 模型、已有 UseCase（如 GetUserUseCase）
- **User 字段**：id、username、email、avatarUrl、createdAt、updatedAt
- **现有 API**：如 GET /api/v1/users/{id}、GET /api/v1/users（是否已有搜索）
- **数据库**：users 表结构、是否已有 username/email 索引
- **发现**：缺搜索接口、缺索引、离线策略需对齐现有 NetworkMonitor 等

### 步骤 3：你的审查

- [ ] 是否准确描述了仓库、模型、API、DB？
- [ ] 是否识别了缺搜索、缺索引等潜在问题？
- [ ] 若无问题，进入规划阶段。

## 阶段二：规划阶段 (Planning)

**目的**：基于研究，得到一份可评审的书面实施计划。

### 步骤 1：发出规划提示词

```
基于 docs/research/001-user-search.md，我要实现用户搜索功能：支持按用户名/邮箱搜索、不区分大小写、
分页每页 20 条、展示头像/用户名/邮箱、处理离线。写一份详细 plan.md，包含实现方案、要改/要建的文件、
关键代码片段、测试策略。保存到 docs/plans/001-user-search.md。
```

### 步骤 2：AI 产出 plan.md 要点（仅列核心）

**API 层**：新增搜索接口

```kotlin
@GET("api/v1/users/search")
suspend fun searchUsers(
    @Query("query") query: String,
    @Query("page") page: Int,
    @Query("pageSize") pageSize: Int = 20
): SearchUsersResponse
```

**数据库**：索引 + 本地搜索

```kotlin
// 迁移中：CREATE INDEX idx_users_username ON users(username); 同上 email
@Query("SELECT * FROM users WHERE ... LIMIT :limit OFFSET :offset")
suspend fun searchUsers(query: String, limit: Int, offset: Int): List<UserEntity>
```

**Repository**：离线优先（先本地再网络、失败用本地）

```kotlin
override fun searchUsers(query: String, page: Int): Flow<Result<List<User>>> = flow {
    val local = userDao.searchUsers(query, 20, page * 20)
    if (networkMonitor.isOnline) {
        try { /* api + insert + emit */ } catch { emit(Result.success(local.map { it.toDomain() })) }
    } else emit(Result.success(local.map { it.toDomain() }))
}
```

**UseCase**：对 query 做 lowercase，调用 repository。

**ViewModel**：StateFlow\<UserSearchUiState\>，Idle/Loading/Success/Error；暴露 search(query)。

**UI**：搜索 Fragment、列表 Adapter、item 布局（头像、用户名、邮箱）。

**文件**：修改 UserApi、UserDao、UserRepositoryImpl、UserRepository、DB 迁移；新增 SearchUsersUseCase、SearchUsersResponse、UserSearchViewModel/Fragment/Adapter 及布局。

## 阶段四：实现阶段 (Implementation)

**目的**：按批准的计划与待办执行，你只做监督与简短纠偏。

### 步骤 1：发出实现提示词

```
按 docs/plans/001-user-search.md 全部实现。每完成一个任务或阶段，在计划里把该项标为已完成。
不要停直到所有任务完成。不要加多余注释或 KDoc，持续跑 lint/编译确保没引入新问题。遵循项目规范（参考 AI_CONTEXT.md、AGENTS.md）。
```

### 步骤 2：实施中你的反馈（简短示例）

- 缺实现：`你没有实现 SearchHistoryRepository。`
- 位置：`SearchUsersResponse 应放在 data/model/，不是 domain/model/。`
- 规范：`UseCase 类名以 UseCase 结尾，和现有约定一致。`
- 功能：`搜索框要加清除按钮，点击清空内容。`

### 步骤 3：完成后检查

- 在计划中确认所有项已勾选。
- 运行 lint、单元测试；必要时跑集成/UI 测试。
- 真机验证：有网/无网下搜索、分页、防抖、历史（若已纳入计划）。

## 流程总览（便于对照）

| 阶段     | 你做的                         | AI 产出 / 你得到         |
|----------|--------------------------------|---------------------------|
| 研究     | 发研究提示词（含强调词）       | research.md               |
| 规划     | 发规划提示词（基于 research）  | plan.md                   |
| 批注     | 打开 plan、加批注、发回「不要实现」 | 更新后的 plan → 待办列表  |
| 实现     | 发「全部实现」+ 持续校验       | 按 plan 打勾并写代码       |
| 纠偏     | 简短指出缺啥/错哪/改哪         | 修正后的代码               |

---

## 关键要点

1. **书面先行**：先有 research.md，再有 plan.md；未认可计划前不写代码。
2. **批注把关**：用内联批注把约束和判断写进计划（如 NOCASE、防抖、历史、测试风格），发回时必须带「不要实现」。
3. **待办再实现**：批注满意后先要待办列表，再进入实现。
4. **实现时少 micromanage**：说「全部实现」后让 AI 按任务跑，用短句纠偏即可。
